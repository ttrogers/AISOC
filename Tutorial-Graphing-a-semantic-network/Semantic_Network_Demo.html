Visualizing the human semantic network for animals in R¶

This page walks through the process of building and visualizing graphs (network data structures) in R, introducing some basic features of R along the way. It is also available as a JuPyteR notebook that you can download by clicking here. Graphs are useful structures for describing many different kinds of data. As an example domain, we will use graphs to visualize mental structures--specifically, the concepts and relations amongst various different animals that exist in your mind!

Background¶

What are graphs?¶

Graphs are data structures that describe networks. They are composed of two elements:

  1. nodes (sometimes called vertices), with each node corresponding to some item
  2. edges (sometimes called links), with each edge connecting two nodes, and representing some relationship between them. Edges can be directed--the relationship goes from node A to node B--or they can be undirected--A and B are linked, with no direction from one to the other.

For instance, consider the network that governs air travel acround the globe. We can view each airport as a node in that network, and each flight from one airport to another as a directed edge. We can represent the whole systems as a big, directed graph, and can use the graph for a variety of purposes--to find the shortest flight from one city to another, to figure out where to increase the frequency of flights, or to find geographic regions within which air-travel is easy or difficult.

Node and edges can possess attributes that describe their properties. Consider the attributes of airports: they vary in size, number of flights, the airlines they serve, the modernity of their facilities, etc. Likewise the flights between airports vary in their properties: how big the plane is, how expensive the ticket, which airline provides the flight, how fast the trip is, and so on. Each such attribute can be stored together with the corresponding node or edge in the graph structure. The attributes can then be used when plotting the graph to govern the size, shape, color, and so on, of the resulting nodes and edges.

Graphs are useful for two reapons. First, they can describe many, many more kinds of datasets. The Web is a giant graph, with each site corresponding to a node and links between sites corresponding to edges. Google uses the structure of the network to generate useful search results. Your social relationships with other people can be viewed as a graph: each person is a node, and the friendships between people correspond to edges. Brains are giant graphs: nodes correspond to neural populations, and edges are the neural connections from one population to another. Second, methods and algorithms have been developed to discover or exploit different aspects of structure in graph--ways of measuring the importance of each node, or for finding the shortest path from one node to another, or of grouping nodes into "communities" that are tightly interconnected with one another and only loosely connected to others, or of assessing how information will spread in a population. Third, graph visualizations can often promote new insights into the structure of the data.

Representing semantic knowledge as a graph¶

In cognitive science it is common to think of concepts in your mind as being related to one another in a semantic network: a kind of graph wher nodes correspond to individual concepts (like "table" or "dog" or "idea"), and edges correspond to links in memory between concepts. By "links in memory" I just mean that thinking of one concept automatically "brings to mind" some other concept. When I say cat, you may automatically think "dog." When I say "ice-cream," you might automatically think "cone" or "yummy" or "fattening." So one way of thinking about how knowledge is represented in your mind is as a graph structure containing the concepts you know as nodes, and as edges, the propensity for one concept to automatically activate another.

The data¶

A common way to measure the nodes and links in some part of your "semantic network" is a simple task called verbal fluency. The participant is given a category name and asked to list all the examples of the category she can think of in whatever order they occur to her. If the category is "animal," the participant just lists all the animals she can think of. The list produced can be viewed as laying bare some of the nodes and edges in her graph. Suppose she begins with "dog." Thinking of dog will automatically bring another concept to mind--say "cat." So when dog precedes cat, we can infer there is a link in the semantic network that goes from dog to cat. Thinking of cat will bring another item to mind--say "bird"--revealing a link from cat to bird, and so on. Any one participant will only produce a small set of nodes and links, but if we gather together data from many participants, we might begin to get a broader picture of the nodes and links amongst animal concepts in a group semantic network.

For this tutorial, I asked people to spend 2 minutes typing out all the animals they could think of, and had them send me these lists as text files. There are 19 lists in the repository where this notebook is found. You will need them to complete the rest of this tutorial.

Making a graph¶

We will use the igraph package loaded earlier to create a graph object. igraph takes a two-column dataframe as input, with the first column labeled "from" and the second column labeled "to." Each row describes a single edge in the graph by indicating the node where the edge originates (from) and the node where it terminates (to). This kind of dataframe is sometimes known as an edge list (even though it is not technically a list in R!). The data can be integers (like, from node 5 to node 10) or strings indicating the node names (like from "mouse" to "hippo"). In our case it is v useful to know the node names, since each node corresponds to a word produced in the fluency task.

So: from the list of words generated, we want to create an edge list. An edge in the graph indicates a link in memory between an item A and the next item B. In the data above, the first item that sprang to mind was "elephant"; producing "elephant" caused the next item ("cat") to come to mind--so in this participant, there is a mental link going from elephant to cat, which we want to represent as a directed edge in the graph. Likewise there is a link from cat to dog, dog to lizard, etc. So the "from" nodes include all of the items in the list except the last, and the "to" nodes include the 2nd item to the last item. We can generate the edge-list dataframe from the raw list like this:

A more interesting graph will collapse the data across responses from all participants. We could iterate the steps above for each, but that is v time consuming. Instead let's write a function to do it for us:

In [11]:
dget.edges <- function(fname){
    #Function to create an edge-list data frame from a verbal fluency list
    #fname: name of text file containing list of items produced, one item per line

    tmp <- read.table(fname, stringsAsFactors = F, sep="\n") #read the file to temporary object
    tmp[,1] <- tolower(tmp[,1]) #Make it all lower-case
    tmp[,1] <- gsub(" ","",tmp[,1]) #Remove any spaces in the names
    
    nitems <- dim(tmp)[1] #Total number of items produced
    
    o <- data.frame(from=tmp[1:(nitems-1),1], to=tmp[2:nitems,1]) #Make edge-list dataframe
    o #Return edge list dataframe
  }

fnames <- list.files(pattern="*.txt") #Get the filenames for all text files in the current directory, store them in fnames
biglist <- dget.edges(fnames[1]) #Read the edgelist for the first textfile, store in biglist
nfiles <- length(fnames)  #How many files are there?

#Loop over all files after the first, read in the edgelist, and bind it to the biglist object
for(i in c(2:nfiles)) biglist <- rbind(biglist, dget.edges(fnames[i]))

dim(biglist)
  1. 559
  2. 2

In R, the for command initiates a loop in which a temporary variable (in this case i) takes on, in turn, each value in a specified vector, then carries out the following command. Here the variable i is iterating over a vector of integers ranging from 2 to the total number of files (nfiles). It first takes the value 2, then carries out the next command: binding together the current biglist and the file read in by dget.edges(fnames[i]). Since i currently equals 2, this will read in the second file in fnames, and bind it to the biglist object. Then i will iterate--it will take on the next value in the integer vector (3), and run the command again: reading in the third edgelist and binding it to the biglist object. Going all the way through the vector will successively add in each individual edgelist to the master edgelist. When the loop completes, all 559 edges across all lists will be stored in a single object.

We could try plotting this master graph, but there is a problem. There may be some edges repeated across subjects. For instance, many subjects say "cat" after "dog," so in the master list there are several different rows indicating the presence of an edge going from cat to dog. While this is a coherent graph, it seems weird to have several different instances of the same edge. Instead, perhaps we want a single edge for such items, plus an edge attribute indicating how often that edge appeared across participants.

In igraph, you can specify the edges of the graph using the syntax E(name of graph object). Likewise you can specify just the vertices (nodes) using: V(name of graph). When nodes or edges have attributes, you can specify the attributes you want using the dollar-sign syntax. For instance V(g)$name will return the "name" attribute for all vertices in graph g. You can use the same syntax to create a new attribute for all vertices or nodes, by just creating a new name for the attribute and assigning it a value. In the above code, I created a new attribute for all edges called "frq" and assigned a value of 1 to this attribute for all edges.

So right now all edges have a frq attribute of 1---but don't forget, edges that are repeated multiple times across lists appear multiple times in the edge list. What we would like to do is to simplify the graph by collapsing all examples of a given edge to a single edge, and adding together the frq value across these instances. So if an edge appears 7 times in the master list, the simplified graph will replace those 7 edges with a single edge, but will give the frq attribute of the new edge a value of 7:

Adding attributes to the nodes¶

There are some data about the items produced we probably want to compile and display in the graph. For instance, how often was each item produced across lists? Maybe we want more frequent items to appear as larger nodes in the plot. And, there are some data that we can get from the graph itself that we may want to display. For instance, how many edges are attached to each node (ie, what is the node degree)? Or we use the graph connectivity structure to group the nodes into clusters of items that are all densely interconnected with one another, and see if this grouping reveals anything about the mental structures that govern our semantic memory. Each of these pieces of data can be stored as an attribute with the nodes in the graph, and can then be used in visualization.

igraph provides tools for automatically computing certain properties of graphs. To calculated the node degree (number of edges attached to each node), use the degree function. There are a variety of algorithms for funding communities (clusters of densely connected nodes) in a graph--for directed graphs like this one, I like the spinglass algorithm.

In [36]:
#To compute the degree of each node and store it as an attribute
V(g)$degree <- degree(g)

#To find communities in the graph using spinglass and store the results in a new object:
gcom <- spinglass.community(g)

#The new gcom objects has lots of information about communities. 
#However the membership subfield is a vector that indicates 
#to which community each node belongs. This can be stored as an attribute
V(g)$membership <- gcom$membership

#You can then use these attributes to set size and color data for the visualization:

nclust <- max(gcom$membership) #How many clusters were found?
V(g)$color <- rainbow(nclust)[gcom$membership] #Set the color of each node using rainbow colors
V(g)$size <- V(g)$degree  #Size the size of each node to show the node degree

I'll plot the results, but be warned, this is messy--we will get nicer visualizations further down.

In [37]:
plot(g, layout=layout_nicely, vertex.label.cex=.6, edge.arrow.size=.2)

Here each circle is an animal name. Circle size shows the node degree, and colors indicate the clusters of nodes (communities) discovered by spinglass. The output is kind of messy though. Perhaps we can simplify the graph further, by removing all the items that were only produced by a single person. To do that, we need to figure out how often each word appeared across all lists. Let's gather all the individual lists into a single master-list, then count how often each word appears in the master list, then store this number as a "frq" attribute for the corresponding node.

In [44]:

Interactive visualization with visNetwork¶

There is! Specifically, the R package visNetwork, loaded earlier using library(visNetwork).

Whereas igraph contains all graph information in a single graph object, visNetwork interprets graph information from two different data frame objects: one containing a description of the nodes in the graph and their attributes, and the other containing a description of the attributes. Luckily, igraph and visNetwork play nicely together, and it is easy to generate one format from the other. To create a visNetwork-style description of the nodes and the edges from the igraph object we created, just do this:

In [56]:
n <- as_data_frame(gsm, what="vertices") #Creates a dataframe from gsm describing the nodes/vertices, stores it as n
l <- as_data_frame(gsm, what="edges") #Creates a dataframe from gsm describing the edges/links, stores it as l

Now we can use visNetwork commands to generate an interactive visualization of the graph. I'm going to give a lengthy command here so you can see, then will unpack it below.

In [67]:
visNetwork(n, l) %>%
           visOptions(highlightNearest = list(enabled = T, hover = T)) %>%
           visPhysics(stabilization = F, solver="forceAtlas2Based", 
                      forceAtlas2Based=list(gravitationalConstant = -2))

This is a static snapshot of the result. Click the image to open a dynamic visualization, of the kind you would see in R itself. In that visualization you can zoom in and out, click and drag the whole graph or individual nodes, and highlight all the items connected to an individual node by hovering over it or clicking it with the mouse. And, visNetwork has some cool algorithms for automically sorting the layout. When you hover over a node, it's "title" attribute pops up in a yellow box. I find it much easier to see what is going on with these abilities to manipulate the graph.

In this case the semantic structure is now clearer. The purple nodes mainly contain farm animals; the blue nodes are mainly water animals; the red nodes are mainly zoo animals, etc. So how do the preceding commands work to generate this image?

In visNetwork, the special operator %>% is a kind of "pipeline." It basically says, take the output of the preceding command and pass it on as the first input argument to the next command. So the above code first runs visNetwork with the n and l dataframes we made as arguments, starts to generate a visualization, but before rendering it the visualization object gets passed to the next command, visOptions. visOptions is a function that sets some options for the visualization--in this case, enabling the "highlight nearest" function when you hover with a mouse. The output of that function in turn is passed forward to the visPhysics function, which controls parameters of the simulated physics in the visualization.

That's right--the visualization is actualy simulating real physics in your browser. The layout of the nodes and edges is determined as follows: visNetwork treats each node as kind of like a little electrical particle, all the same charge. The closer two nodes are together, the more they repel each other. The edges are treated as little springs--the further apart two connected nodes get, the more the spring tries to pull them back together. visNetwork starts everything off in a random position, then simulates the resulting physics. Nodes that are all connected to one another have many "springs" joining them, so get pulled toward one another despite the repulsion. Nodes that are connected by few springs get pushed apart. The whole graph self-organizes into a visualization where nodes in the same community tend to be laid out in similar locations.

The arguments in visPhysics basically say: use an algorithm called "force Atlas 2" to simulate the physics, and set the gravitational constant (amount of repulsion) to -2. The more negative that nubmber gets, the stronger the repulsion, and the further apart the nodes will be.

visNetwork has many alternative ways of laying out graphs, including all of the layouts available from igraph. Check out the visNetwork guide to read more.
